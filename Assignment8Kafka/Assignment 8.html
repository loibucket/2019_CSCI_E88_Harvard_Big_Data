    Assignment 8: Messaging Tier with Kafka

    Please use the following template for your submission: Assignment8_Submission Template.docx

    In this assignment, we are trying to build another part of a Big Data pipeline - collection tier plus messaging tier.

    For a 3-node Kafka cluster setup:

    • we provide full instructions on how to do this using Docker compose - in the Lab
    • you can do the same on AWS EC2 nodes instead - it is not a very straightforward process, so here are a few pointers for that setup:
      • make sure each node has a distinct broker.id, has privateIPs to talk to Zookeeper, but uses publicIPs as advertised hosts in the server.properties for each node
      • myid file for zookeeper should have the same id as the broker.id
    • you can elect to run a 1-node Kafka cluster installed locally on your laptop - the easiest option - but you will be deducted 20 points from your total HW points

     

    Problem 1: [Points: 20] Basic Kafka setup

    • Create a 3-node Kafka cluster using docker compose ( as demoed in the Lab)
    • Bring it up using --scale kafka=3
    • Connect to Running kafka container-1 using `kafka exec -it container-1name bash` command
    • Once you are connected to the running kafka container issue kafka-topics --describe command and verify there are no topics in this cluster yet
    • Create a topic "problem1" with 2 partitions and replication factor = 2
    • Describe the cluster and show its state via kafka-topics --describe command
    • Start a kafka console consumer in a new terminal and leave it running 

    docker exec  kafka_kafka_1 kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic problem1

    • Open another terminal and start kafka console producer in an interactive mode and try sending some messages. Messages sent in producer should show up in console consumer

    docker exec  -it kafka_kafka_1 kafka-console-producer.sh --bootstrap-server localhost:9092 --topic problem1

    • Examine kafka commit logs, by using the kafka.tools.DumpLogSegments tool , for each partition of the "problem1" topic

     

    Problem 2: [Points: 25] Flume + Kafka integration 

    • Use your Web server + Flume setup you did for HW5 Problem 2 [ 'tail -f' source, memory or file channel] 
    • Create a new topic, "problem2" , with 2 partitions
    • Connect to Running kafka container-1 using `kafka exec -it container-1name bash` command
    • Run `docker ps -a` to look at running containers and get the local port that is forwarded to kafka-1 9092 port
      • For example 0.0.0.0:32781->9092/tcp) means that local port is 32781
    • Add Kafka Sink to your Flume - to push events into the "problem2" topic 
    • Make sure your flume kafka sink points to localhost:32781
    • Start your event/request generation script developed in HW5/P2 to generate CURL requests to the web app  [or however you were generating load to your web server]
    • Start a Kafka console consumer for "problem2" topic - verify you are getting your web server logs
    • Add Flume's UUID Interceptor - to add UUID to each event
    • Verify you are getting events with UUIDs now in Kafka [hint: UUIDs are not going to be added to the event payload, but rather to event headers ….]

    ./bin/kafka-console-consumer.sh --new-consumer --bootstrap-server kafka-1:9092 --topic topic1 --property print.key=true --property key.separator="-"

     

    Problem 3 [Points: 30]: Programming with Kafka APIs

    • Create a new topic, "problem3",  with 3 partitions
    • Using Kafka API, create a new Problem3Producer application that generates and sends events to the "problem3" topic - generate the same type of log events we've worked with so far, plus add a UUID:
      • one event is one line in the format: <uuid> <timestamp> <url> <userId>
      • make sure you have at least 3 different userIDs
    • Verify your producer works correctly by receiving generated events via Kafka console consumer
    • Using Kafka APIs, create a Problem3Consumer (single-threaded is OK), with the "p3consumer" consumer group.id,  that listens to the "problem3" topic - demonstrate that it receives events generated by the Problem3Producer
    • For each event received, the Problem3Consumer should print out its offset, partition number, event key and event body
    • Demonstrate that the Problem3Consumer receives events from all 3 partitions and they are equally distributed

     

    Problem 4: [Points: 25] Kafka consumer groups 

    • Building up on Problem 3's setup: start a second instance of the Problem3Consumer, with the same consumer group ID , 'p3consumer'
    • Demonstrate how partitions are re-balanced between the two consumer instances - which partitions are processed by which instance now?
    • Start two more instances of the Problem3Consumer with the same group ID - to the total of 4 instances now. Demonstrate the partition re-balancing and explain the results. Which instances are processing which partitions, if any?
    • Stop two instances of the Problem3Consumer - demonstrate the partition re-balancing again

     

    Bonus Problem A: [+10] partitioning by userID

    • Create a new ProblemAProducer, by modifying Problem3Producer, to send events to Kafka partitioned based on the userIDs: 
      • Events with the same userID should go to the same partition number
      • Make sure that there is at least one userID corresponding to each partition (you have 3 partitions, remember)
    • Start your  Problem3Consumer and verify that the events for the same userID are received from the same partition

     

    Bonus Problem B: [+15] ISR or not ISR?

    • use the same 3-node Kafka cluster setup (either in AWS or Docker)
    • Create a topic, "problemb", with 4 partitions and replication factor 2
    • Examine the state of the cluster and show how all replicas are spread over the cluster nodes - show all ISRs
    • Using console producer send a few events to the "problemb" topic; verify you have received them using the console consumer
    • Stop Kafka on one of the nodes, wait for a few seconds - show the state of the cluster now, and all ISRs of the "problemb" topic
    • Run console producer and consumer again, see if you can still send/receive messages
    • re-start Kafka on that node - observe how fast the cluster gets into the full ISR state
