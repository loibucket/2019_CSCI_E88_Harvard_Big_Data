    Assignment 10: Data Modeling and Processing with Cassandra

    Please use the following template for your solution document: Assignment10_SubmissionTemplate.docx

     

    Problem 1: [Points: 25] data modeling with Cassandra

    • Install Cassandra (either locally or on AWS/Docker) and create a keyspace named “hw10” using CQLSH
    • Given events in the usual format <uuid> <timestamp> <url> <userId> <country> <ua_browser><ua_os> <response_status> <TTFB>:
      • design a table, called "hw10_p1" in Cassandra to store individual events and be able to answer the following two queries:
      • Query 1: Count of events per country per URL for a specified time range [t1 … t2]
      • Query 2: Average TTFB per country per URL for a specified time range [t1 … t2]
    • Explain the design of your table's Primary Key.
    • Using CQLSH insert a few events, spanning three or more hours of time
    • Create CQLSH queries for the above Q1 and Q2 , run them for a few variations of input parameters (URLs, countries, time ranges), demonstrate the results

     

    Problem 2: [Points: 25] Range queries with Cassandra

    • Assume that you are getting too many events for each URL+country combination now - millions and millions per day; modify your table from Problem1 by adding partitioning by hour  - call it "hw10_p2"
    • Explain the design of your table's Primary Key
    • Using CQLSH create the table, insert a few events, spanning three or more hours of time
    • Create CQLSH queries for the above Q1 and Q2 , run them for a few variations of input parameters (URLs, countries, time ranges), demonstrate the results
    • Explain the main difference between "hw10_p1" and "hw10_p2" tables

     

    Problem 3: [Points: 30] Programming with Cassandra drivers

    • Instead of generating/inserting events manually into your "hw10_p2" table you created for Problem2 - write a small program that performs the following:
    • Generate a few hundred events in the usual format, spanning a few hours in time
    • Using Cassandra driver, insert these events into the "hw10_p2" table - for example, for Java, you could use this driver: https://docs.datastax.com/en/developer/java-driver/4.3/ , For Python: http://datastax.github.io/python-driver/getting_started.html )
    • Also using the driver, implement the Q1 and Q2 queries and print out the results

     

    Problem 4: [Points: 20] data consistency with Cassandra

    • Modify your app from Problem 3 to use "async" operations writing events into Cassandra (for Java: https://docs.datastax.com/en/developer/java-driver/4.3/manual/async/ , for Python: http://datastax.github.io/python-driver/getting_started.html#asynchronous-queries )
    • Use QOURUM consistency level for your writes and reads
    • Run the Q1 and Q2 queries only after you have ensured that all write requests are completed

     

    Problem 5: [Bonus: +20] Spark Streaming and Real-Time Views with Cassandra

    • Create a Spark Streaming job to read in events in the usual log format
      • you can use Kafka as your input, and re-use your Kafka+Spark setup from the HW9
    • Implement a job that counts:
      •  number of clicks per URL per batch [you can pick whatever batch duration you prefer]
      • and AVG TTFB per URL per batch
      • prints the results after each batch
      • stores data into a new Cassandra table, "hw10_p5"
    • for Java, you can use this connector: https://github.com/datastax/spark-cassandra-connector
    • Design the "hw10_p5" table to be capable of answering the following questions:
      • Q1: Get count of events per URL for a specified time range [t1 … t2]
      • Q2: Get an average TTFB per URL for a specified time range [t1 … t2]
    • Using CQLSH, create the table
    • Create CQLSH queries for the above Q1 and Q2 , run them for a few variations of input parameters (URLs and time ranges), demonstrate the results
    • Explain the main difference between "hw10_p2" and "hw10_p5"

    Problem 6: [Bonus: +15] Cassandra cluster

    • Create a 3-node Cassandra cluster in 2 data centers - either on AWS or using Docker/VMs
    • Create the "hw10" keyspace with the replication factor 2 in one data center and replication factor 1 in another
    • Create the same table, "hw10_p2", from Problem 2 and insert a few events 
    • Using 'nodetool' utility inspect:
      • Status of the cluster
      • Which nodes hold some specific row (by partition key, pick one that you inserted into your table)
